# Changelog

## Unreleased

### 标签页：同一个飞书聊天并行多任务

- 新增网关层 `/tab` 命令族：`/tab new` 新建标签页、`/tab <1-9>` 切换、`/tab` 查看列表。每个标签页是一个独立的虚拟 surface（`<surface>#tabN`），拥有自己的工作区绑定、会话与消息队列，多个标签页的任务真正并发执行。
- 切换标签页不会打断其他标签页正在执行的任务：进行中的过程卡与最终结果照常回复在各自源消息下；卡片按钮、表情回应按消息级映射路由回所属标签页，与当前所在标签页无关。
- `/tab new`（以及切到从未用过的编号）会自动在新标签页弹出工作区选择卡，可直接接入已有目录或导入 Git 仓库。
- 标签页指针持久化在 `StateDir/feishu-tabs-<gateway>.json`，daemon 重启后保持；存量会话（标签页 1 = 原 surface）不受升级影响。
- `/tab` 完全在飞书网关层实现与拦截，核心 orchestrator 无改动。

## v1.6.0

覆盖范围：自 `v1.5.0` 以来的 301 个提交（按提交统计，不按 issue）。

### 在飞书里继续“当前正在跑的工作”更自然了

- 现在你可以直接回复当前正在执行的那条用户消息，把补充说明直接 steer 进本轮执行，不必等它跑完再补开新一轮。
- 排队中的文字仍然可以用 `ThumbsUp` 升级成 follow-up；如果队列里已经攒了多条，也可以直接用 `/steerall` 一次性并入当前 running turn。
- `/compact` 现在成为正式可用的手动上下文整理入口，需要主动压缩当前 thread 时不必再绕路。
- `/history` 历史卡支持分页和单轮详情来回切换；刚接管工作区但还没显式建新会话时，直接发文字也能更自然地进入新 thread。

### 工作区、文件和目标选择体验明显更顺了

- `/list`、`/use`、`/useall` 已经进一步收敛成统一目标选择体验，可以更直接地切工作区、挑会话或新建会话。
- 现在可以直接在目标卡里“添加工作区”：既能接入已有本地目录，也能直接填 Git 仓库地址，把仓库拉下来并接成新工作区。
- `/sendfile` 可以从当前工作区里直接挑文件发回飞书聊天，分享本地产物不必再手工找路径。
- 目录/文件选择卡统一成了下拉式路径选择，手机端和长目录场景下更稳、也更不容易误触。

### 过程信息和最终结果更容易看懂了

- 飞书里现在会显示更完整的计划更新、命令执行、工具调用、Web 搜索和探索进度卡，执行过程不再只是零散文本。
- 最终答复卡补上了更清楚的 token usage、thread usage 和 context left 信息，长任务是否“快撞上下文墙”更容易判断。
- 最终卡在正文过长时会自动拆分；本地 `.md` 和单文件 `.html` 链接的预览替换也更稳，失败时不会把整段 Markdown 搞坏。

### 定时任务正式进入可持续使用状态

- `/cron` 现在不只是“打开一个表”，而是一整套可持续用的定时任务工作流：`status / list / edit / reload / repair` 都已经补齐。
- 定时任务既可以直接跑本地工作区，也可以按 Git 仓库地址和 ref 拉取临时 worktree 后执行。
- reload 结果、运行记录、配置入口，以及时区/分钟精度展示都更完整，日常排查任务有没有生效更直接。

### 安装、升级和多实例部署更成熟了

- 新增 Windows 在线安装脚本，Windows 用户不再需要完全手工走 release 包安装。
- 升级入口统一收敛到 `/upgrade` 体系，`latest / track / local` 语义更清楚；`production / beta / alpha / dev-latest` 这几条发布轨道也更完整了。
- 本地升级、嵌入式 shim、多实例 `systemd --user`、工作区绑定实例和 repo 绑定升级路径都更稳，更适合一台机器跑多套环境。
- Docker 部署入口已经移除，当前默认推荐直接用单二进制本机安装。

### 管理页和预览链路更稳了

- 管理页入口统一到了 `/admin`，配套跳转和调试入口更一致。
- 网页预览壳子、授权缓存、文件上传和清理链路都补了稳定性修复，外部访问与预览链接更少出现“点得开但用不住”的状态。
- 状态卡会展示更多当前实例信息，例如当前 Git 分支和权限缺口，自查成本更低。

## v1.5.0

覆盖范围：自 `v1.4.0` 以来的用户可感知变化。

### 默认使用路径更明确了

- 现在更推荐直接在飞书里按工作区和已有会话继续当前工作，这是大多数场景下更直接的起点。
- 只有在你明确需要让飞书跟着编辑器当前焦点变化时，才再接入 VS Code；VS Code 更适合作为按需联动，而不是开始使用前的默认前提。
- 新的 WebSetup 流程把环境检查、飞书扫码接入、自动启动和按需的 VS Code 接入整理成了更清晰的顺序，首次配置时更容易知道下一步做什么。
- 如果你是新建飞书应用，现在可以直接通过扫码完成接入；如果是已有应用，也能在同一套流程里继续补齐权限、事件、菜单和回调。
- 当前用户升级入口也更明确了：直接在飞书里使用 `/upgrade latest` 即可检查、开始或继续升级。

### 在飞书里继续当前工作更顺了

- `/menu`、`/help` 和一组参数卡已经整理成同一套入口，不用记太多命令也能找到常用操作。
- 同一上下文里的卡片切换会尽量原地更新，减少不断堆出新卡片的情况。
- 回复、图片结果和部分工具结果的回显方式更完整，在群聊或连续对话里更容易看懂上下文。

### 多工作区和多会话切换更清楚了

- normal mode 现在更明确地以工作区为中心，`/list`、`/use` 和 `/useall` 在多项目场景下更容易看出自己正在接管哪里。
- 近期会话和持久化线程会更自然地出现在选择列表里，重回之前的工作上下文更直接。
- VS Code 和飞书之间的会话切换规则更一致，误切到错误工作区或错误线程的情况更少。

### 重启、恢复和长任务的连续性更稳了

- daemon 重启后的恢复线索会保留下来，恢复会在合适时机继续，不容易因为一次中断就丢掉上下文。
- 可选的 `autowhip` 会在回复明显未完成或上游短暂波动时继续补做，并把重试或结束状态说明清楚。
- 最终回复会补充更多完成状态信息，等待和恢复过程也更容易判断。

### 管理页和外部访问更易用了

- 管理页在手机和窄屏下的布局更稳，首次配置和日常检查不必强依赖桌面宽屏。
- 新增了更完整的外部访问基础能力和调试入口，在需要从本机外打开管理页时更容易自查。
- 部分状态与预览信息改成更贴近远端实际运行情况的展示方式，减少“页面看起来正常，但远端并没有准备好”的判断误差。

## v1.4.0

覆盖范围：自 `v1.3.0` 以来的用户可感知变化。

### 安装和开始使用更顺了

- 现在可以直接按 `production / beta / alpha` 三条 track 安装，想尝鲜或回到正式版都更直接。
- macOS 安装链路补了兼容性修复，在线安装脚本在空参数场景下也不会再意外报错。
- WebSetup 的引导文案、回调说明和 VS Code 接入说明重新整理过，首次配置时更容易按顺序走通。

### 飞书里更像“继续当前工作”，而不是“重新找上下文”

- `/help`、`menu` 和 `/menu` 现在可以直接把当前可用命令和快捷入口展示出来，不用再靠记忆。
- 最终回复会直接回在你触发它的那条消息下面，群聊里更容易看懂上下文是谁发起的。
- 系统卡片里对代码片段和行内 token 的渲染更清楚，最终卡片标题也会带一点原始提问预览。
- 如果一条文字还在排队，你可以给这条消息点个赞，把它升级成对当前正在执行回复的跟进或纠偏。

### 旧卡片和旧动作现在会更明确地“失效”，减少误触

- daemon 重启后，属于上一个生命周期的旧消息、旧菜单动作和旧卡片会被统一识别并明确提示，不再悄悄落到当前流程里。
- 旧的 `/newinstance`、`/killinstance` 和历史恢复卡片已经退出主流程；看到迁移提示时，直接改用 `/use`、`/useall` 或 `/detach` 即可。
- 旧卡片如果已经过期，会直接告诉你重新发哪条命令拿新卡片，而不是继续沿用一张已经失真的按钮卡。

### 恢复和稳定性更好了

- daemon 重启后的后台恢复流程更稳：恢复线索会持久化，恢复会延后到合适时机再进行，不会一上来就误判失败。
- `/detach` 现在不仅能断开接管，也能顺手取消当前还没完成的恢复流程，逃生路径更明确。
- 本地链路波动时，正在执行的 turn 更不容易被中途打断；队列余量和停机通知也都更完整。
- 管理页里不再把纯后台恢复实例暴露成普通可接管对象，日常视图更聚焦。

### 升级和回滚基础补齐了

- 安装状态现在会记录更完整的版本与来源信息，为后续升级检查、候选调试和自动回滚打下基础。
- 已经补上 helper 驱动的升级回滚流程，切换版本时对正在运行的 wrapper 和 daemon 处理更稳。
